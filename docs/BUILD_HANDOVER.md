## 1. What you are building (one paragraph)

TenderScore mechanises the moderation pack for UK public sector tender evaluation. A buyer locks a published framework (criteria, sub-criteria, lots, band descriptors, weightings, gates). Bidder submissions are ingested, split per question, hashed, scanned for prompt injection and anonymised. For each question and each bidder *in isolation*, the scoring engine runs multiple passes at temperature 0 against a pinned model version, producing a recommended score that must cite verbatim spans from the bidder's own answer; citations are validated in code, and score variance routes the result to human moderation. Named humans confirm or amend every recommendation with a logged rationale. Every state change is written to an append-only, hash-chained audit store, from which moderation packs (and later, standstill/assessment summaries) are generated by construction. **AI scores, humans moderate, AI documents.** This build covers the Evaluate tier only; Tier 2 ("Assure") modules are designed but not built.

## 2. Non-negotiables (the short list — `CLAUDE.md` is authoritative)

- **Audit trail:** append-only, hash-chained, DB-enforced (no UPDATE/DELETE). No state change without an `AuditEvent`. Enforced in middleware, not per-endpoint.
- **Determinism:** temperature 0 only in scoring; model version pinned per procurement at lock time and recorded on every pass; prompts are versioned, hashed registry artefacts — never inline strings; all inputs content-hashed; runs replayable from the record.
- **Isolation:** schema-per-tenant; one bidder, one question per scoring call; comparative-in-prompt scoring is permanently prohibited.
- **Instruction/content separation:** bidder text is data, never instruction; structurally separated, injection-scanned, truncated at the published limit; gateway taint-checks that content never reaches the instruction layer.
- **Citations or nothing:** every justification claim cites a verbatim span; validated by code against the hashed source; invalid citation → pass rejected, rerun, flagged.
- **Variance routing:** 3 passes (5 for high-weighted criteria); within one band → converged; beyond one band → escalate to human, never auto-recommend.
- **Safety never paywalled; no cross-tenant learning; no training on customer data.**
- **Permanently excluded — do not build, not even a stub:** automated award decisions; performance/reputation scoring; AI-authorship detection or penalties; evaluation against unpublished criteria; single-prompt comparative scoring.

## 3. Execution plan (milestones — full detail in architecture Part I)

Build in this exact order. Each milestone's "report on stop" is what you hand back before proceeding.

**M0 — Scaffold.** Monorepo per architecture Part C; Docker Compose (`db` postgres 16, `minio`, `api`, `worker`, `web`); CI (ruff, mypy --strict, pytest, vitest, axe-core); Makefile (`dev`, `test`, `lint`, `regression`, `redteam` — last two may be loud-failing stubs); FastAPI app factory + health endpoint; React+TS+Vite shell; ADR-001 recording the stack. *No business logic.*
→ **Report:** `make dev` boots all services; `make test` green; health endpoint returns; CI runs on a PR.

**M1 — Tenancy, auth, audit spine.** Schema-per-tenant provisioning; dev `IdentityProvider` (email+password+TOTP) behind the interface; RBAC roles + the distinct anonymisation-map privilege; audit middleware; append-only hash-chained `audit_events` with REVOKE + trigger enforcement; chain verifier.
→ **Report:** audit-completeness test passes; demonstrate UPDATE/DELETE on `audit_events` is rejected at the DB level; verifier detects a tampered chain in a test.

**M2 — Ingestion.** Upload to object storage; PDF/DOCX parse; split into question responses; content hashing; deterministic injection pattern scan.
→ **Report:** a synthetic submission round-trips; hashes stable across re-runs; seeded injection patterns are flagged.

**M3 — Framework service.** Procurements; **lots** and **criteria tree (sub-criteria) as first-class**; band descriptors (verbatim, locked); gates and limits; framework lock (hash of the full tree + descriptors + pinned model version); LLM-assisted extraction *draft* that a human edits before locking.
→ **Report:** post-lock edits are rejected and audited; a locked framework reproduces its hash.

**M4 — LLM gateway + prompt registry.** Gateway per architecture Part E; `AnthropicAdapter`; registry load/reconcile on startup (changed file without version bump fails startup); model-version guard; instruction-taint check; token accounting; every call audited.
→ **Report:** unregistered prompt, wrong model version, and tainted-instruction calls are each rejected with audited refusals; a registered call succeeds and is recorded.

**M5 — Scoring engine.** Orchestrator jobs (per question × bidder); multi-pass at temp 0; deterministic validation incl. the citation matcher; variance routing; calibration gate.
→ **Report:** a full synthetic procurement scores end-to-end; **replay from the audit record reproduces identical validated outputs**; citation validity ≥ 99.5% on the regression suite.

**M6 — Compliance + anonymisation.** Word/page-limit checks; attachment and caveat checks; gate enforcement (a failed gate blocks scoring of that bidder on that criterion); NER+rules anonymiser; token mapping; privileged reveal with per-access audit.
→ **Report:** gate failures behave correctly; anonymised text contains no seeded identifiers; every map access is audited.

**M7 — Moderation workflow + UI.** Tiered queue (converged → moderate → escalated); confirm/amend with **mandatory rationale on amend**; side-by-side descriptor view; evaluator/moderator UI.
→ **Report:** a moderator completes a full synthetic moderation; every decision audited; axe-core clean and a keyboard-only run-through works (WCAG 2.2 AA).

**M8 — Document generation.** Moderation pack (docx/pdf) generated *only* from the moderation record; versioned, stored, hashed.
→ **Report:** pack content is provably derived from record fields (no free-text injection of un-recorded content).

**M9 — Red-team + regression hardening.** Expand the injection corpus to ≥50 variants; order-sensitivity tests; descriptor-vocabulary tuning.
→ **Report:** red-team suite green (zero score-shifting or instruction-altering successes); documented variance distribution.

**M10 — Pre-pilot hardening.** Rate limiting; session hardening; backup/restore runbook; log scrubbing; pen-test prep checklist; OWASP ASVS L2 self-assessment.
→ **Report:** runbooks exist; ASVS L2 self-assessment documented.

After M8 the Evaluate tier is functionally complete on synthetic data. **Do not begin any Tier 2 / Assure module** (consistency engine, claims verification, commercial/ALT, clarifications, standstill generation, Submission Integrity Analysis, SME reporting) — those are P7 and out of scope here. Leave migration headroom so they aren't precluded.

## 5. Things you must not decide yourself (surface them, don't resolve)

These are human-owned (handover Part 6). Where one blocks you, make the smallest safe labelled assumption and flag it; never quietly resolve one.

- **Cloud provider (AWS vs Azure)** — open. Do **not** add Terraform or any provider-specific config. Docker Compose is the only infrastructure. Keep storage (S3 API), KMS, queue and identity behind interfaces.
- **SSO tenant (Entra ID)** — production target; build the `IdentityProvider` interface and a dev provider only.
- **Pricing, naming, design partners, framework/route-to-market** — not engineering concerns; ignore unless they surface in copy, in which case use the placeholder "TenderScore" and flag.
- **Retention defaults** — design the field configurable; default 7 years but mark `[[LEGAL REVIEW NEEDED]]`.
- **Current model version string and UK/EEA inference residency options** — confirm the exact pinned model string and residency configuration at build time rather than hard-coding; if unavailable, use a clearly-labelled placeholder constant in one config location.

## 6. The synthetic oracle (you need this for M5, and only a human can author it)

The regression suite needs `tests/regression/fixtures/synthetic_tender_01/` — one full fictional procurement (district council, Grounds Maintenance Services, 2 lots, 6 quality criteria incl. sub-criteria, 0–5 descriptors, one safety gate at min score 2, 60/40 quality/price), 4 fictional bidders (strong/mid/weak/gate-failer, one seeded with 5 escalating injection attempts), **with hand-authored expected scores and rationales as the scoring oracle.**

You can generate the fictional tender and bidder responses. You **cannot** author the authoritative expected scores — that is the founder's domain expertise and the calibration anchor. Build everything around a placeholder oracle so M5 can run, and flag clearly: `[[HUMAN INPUT NEEDED: Felix to author/review the oracle scores for synthetic_tender_01 — this is the regression ground truth]]`. All content invented; no resemblance to any real bid (Hard Rule 7).
